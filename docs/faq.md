# Frequently Asked Questions ❓

## Agent & Configuration

### Q: Why not use adaptive sampling or on-host ML?
**A:** This toolkit deliberately focuses on optimizations achievable *solely* through static configuration of the standard New Relic Infrastructure Agent (`newrelic-infra.yml`). This ensures broad applicability without requiring agent forks, sidecars, or complex runtime components. Future versions might explore adaptive techniques if they become standard agent features.

### Q: What's the recommended sample rate?
**A:** The Infrastructure Agent defaults to 20 seconds. This toolkit often recommends **60 seconds** (`metrics_process_sample_rate: 60`) as a starting point, providing a ~67% reduction in ProcessSample volume with typically minimal impact on essential visibility. Rates up to 180s or 300s offer higher savings but require careful validation using the `lint` command and NRDB checks.

### Q: How do filters work?
**A:** The agent uses `exclude_matching_metrics` with glob patterns based on the `process.DISPLAYNAME.*` metric format. `true` excludes matching processes. See `templates/filter-definitions.yml` for examples. The `lint` command helps identify if critical processes (Tier-1) might be accidentally filtered.

### Q: Where should I save generated configurations?
**A:** Use the `--output` (`-o`) flag with commands like `wizard`, `generate-config`, and `recommend` to specify a file path (e.g., `-o my-config.yml`). **Avoid accepting the default output path** (`./config/newrelic-infra.yml`) as it will overwrite the example file included in the repository source.

## CLI Usage

### Q: How is cost estimated (`estimate-cost` command)?
**A:**
1.  **Static Model:** Uses heuristics (default 150 processes, 450 bytes/event) scaled by sample rate and an estimated keep ratio (based on filter complexity, default ~0.9 for standard, ~0.4 for aggressive). Provides a baseline estimate.
2.  **Dynamic Model (requires API Key):** Queries NRDB for `bytecountestimate()` histogram data over the specified `--window`. Calculates average event size and observed process count.
3.  **Blended Model (default if API Key provided):** Combines static and dynamic estimates, weighted by the confidence derived from the histogram data quality. Aims for ±10% accuracy against `NrConsumption` data.

### Q: Can I add my own configuration presets?
**A:** Yes. Presets simplify generating common configurations.
1.  Edit the `templates/wizard-presets.yml` file.
2.  Add a new entry under the `presets:` key, following the existing structure:

    ```yaml
    presets:
      # ... existing presets ...
      my_custom_preset:
        description: "My specific setup (e.g., high rate, specific filter)"
        sample_rate: 180  # e.g., 180 seconds
        filter_type: "standard" # or "aggressive", "minimal", or a custom filter name
        overrides: # Optional: specific YAML keys to add/override
          collect_command_line: false
          exclude_matching_metrics:
            process.my_critical_app.*: false # Ensure this isn't filtered
            process.noisy_utility.*: true # Explicitly filter this
    ```
3.  You can then use it with the CLI:
    ```bash
    process-lab generate-config --preset my_custom_preset -o my-preset-config.yml
    # or
    process-lab wizard --preset my_custom_preset -o my-preset-config.yml
    ```

### Q: Can I add custom filter types?
**A:** Yes. Edit `templates/filter-definitions.yml`:
    ```yaml
    filters:
      # ... existing filters ...
      my_filter_set:
        description: "Custom filter excluding chat apps"
        patterns:
          process.slack.*: true
          process.teams.*: true
          process.discord.*: true
          # ... other patterns ...
    ```
Then use `--filter-type my_filter_set` in `generate-config` or `wizard`.
### Q: Why are `validate` and `visualize` commands stubbed?
**A:** The `validate` command (comparing predicted vs. actual NRDB cost post-deployment) and `visualize` (generating charts) were part of an older design iteration and are not fully implemented in the current CLI structure.

*   The `validate` command is a placeholder for future functionality that would automatically compare predicted costs and coverage against actual NRDB data after deployment. For now, use the NRQL queries in `nrdb_analysis/README.md` for manual validation.
*   The `visualize` command (generating charts) is not implemented. You can use the dashboard JSON provided in `nrdb_analysis/` or build custom dashboards in New Relic.

## Development & Testing

### Q: How do I run the tests?
**A:**
```bash
# Ensure dev dependencies are installed
pip install -e ".[dev]"

# Run all tests via tox (recommended, handles environments)
tox -p auto

# Run specific test types
tox -e unit
tox -e integration
tox -e lint
tox -e mutation
```

### Q: What are the quality gates?
**A:** Pull requests generally require:

*   Passing ruff linting and mypy type checking.
*   Passing all pytest tests (unit, integration, contract, property).
*   Maintaining high test coverage (typically >90%, measured by pytest-cov).
*   Low survival rate for mutation testing (typically <10%, measured by cosmic-ray).
